現在の重点事項

実装済みと残課題を区別します。

複数送り先を管理

医院・患者ごとに複数の送り先と既定値を持ち、所有者を実FKで保証します。

使用中は削除不可

進行中注文が参照する送り先はDBトリガーで論理削除を拒否します。

注文時点を保存

選択した送り先を注文スナップショットへコピーし、マスタ変更後も履歴を不変に保ちます。

設計原則

正本とポータル投影を混在させません。

  • 商品・注文・決済・返金・定期購入の将来正本はShopify
  • 医院・患者・活動情報の将来CRM正本はSalesforce
  • 医院内の受け取り準備状態はポータルが管理
  • 外部受信はWebhook Inboxへ先に永続化
  • 外部送信はOutboxで再試行可能にする
  • 決済状態と医院業務状態を別軸で保持
  • 架空データや保存されていない成功表示をしない
  • 患者注文と医院仕入注文を統合しない
別ドメイン： clinic_orders はBGJから医院へのB2B仕入・売上履歴です。患者注文の patient_orders 系列には統合しません。

DB遷移計画

現在の内部注文IDを維持したまま段階移行します。

  1. Phase 0完了

    patient_orders / patient_order_items

    内部注文・明細・冪等性・商品スナップショットを運用。

    現行IDを維持する。
  2. Phase 1-1次に実施

    患者削除方式

    物理削除を無効化・匿名化へ変更し、注文FKのCASCADEを見直す。

    注文履歴を消さない。
  3. Phase 1-2一部完了

    patient_order_events

    状態変更、操作者、変更前後を記録済み。変更理由の記録は今後追加する。

    注文更新と同一トランザクション。
  4. Phase 1-3次に実施

    patient_orders.version

    楽観的排他を導入する。

    競合時は409。
  5. Phase 1-4完了

    delivery_destinations / order_delivery_destinations

    医院・患者の複数送り先と注文時スナップショットを運用する。

    進行中注文の送り先は論理削除不可。
  6. Phase 2-1仕様確定後

    commerce_accounts / Inbox / Outbox

    Shopify接続・Webhook受信・外部送信を分離する。

    外部仕様確定前に作り込まない。
  7. Phase 2-2仕様確定後

    決済・返金・配送投影

    Shopify状態を医院業務状態と別軸で保持する。

    確定額と参考額を混同しない。
  8. Phase 2-3仕様確定後

    patient_subscriptions

    定期購入契約を読み取り用投影として追加する。

    変更・解約はShopify正本。
  9. Phase 3仕様確定後

    Salesforce外部ID対応

    医院・患者・活動の同期方向を定義する。

    CRM正本はSalesforce。
